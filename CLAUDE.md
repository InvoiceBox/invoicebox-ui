# CLAUDE.md

Руководство по работе с этим репозиторием.

## Что это

`@invoicebox/ui` — **опубликованная npm-библиотека React-компонентов** (UI-кит), не приложение. Нет роутов, нет глобального стора, нет SSR, нет собственных сетевых запросов. Потребители устанавливают её и рендерят компоненты внутри своего приложения. ~41 компонент (`src/components/form/`, `src/components/common/`), 6 общих хуков, styled-components 6 для стилизации, строгий режим TypeScript.

Полный аудит перед рефакторингом (2026-07-01) лежит в [audit/](audit/) — сначала прочитайте [audit/README.md](audit/README.md), если делаете что-то помимо мелкого фикса. Любой рефакторинг крупнее тривиального багфикса должен следовать плану волнами из [audit/10-REFACTOR-PLAN.md](audit/10-REFACTOR-PLAN.md) и получать согласование перед началом волны.

## Команды

- `npm test` — запускает jest. **Оговорка:** пока не существует `jest.config.js`, поэтому если `dist/` был собран локально, один и тот же тестовый файл подхватывается 3× (`src/`, `dist/`, `dist/esm/`). Это не баг приложения, а пробел в конфиге — см. [audit/08-tests.md](audit/08-tests.md).
- `npm run storybook` — локальная витрина компонентов, `storybook dev -p 6006`.
- `npm run release` — запускает `release.sh`: собирает dist (CJS + ESM), собирает Storybook в `docs/`, коммитит `docs/`, поднимает версию, мёржит `develop` → `main`, публикует в npm. **Без гейта CI** — выполняет то, что есть в рабочей копии, без единой автоматической проверки lint/тестов/typecheck до этого.
- Сейчас нет скриптов `lint`, `build` или `typecheck` — `.eslintrc.json` и `tsconfig.json` существуют, но пока не подключены ни к одному скрипту (отслеживается в аудите, Волна 0).

## Неочевидные контракты — прочитать перед изменением конфигов сборки/релиза/тестов

- **`docs/` — это сборка Storybook, а не документация.** `release.sh` выполняет `rm -rf dist docs` при каждом релизе. Никогда не кладите туда то, что должно сохраняться — именно поэтому аудит лежит в `audit/`, а не в `docs/`.
- **`dist/` в gitignore и никогда не коммитится** (правильно — проверяйте, что `git ls-files dist` остаётся пустым). **`docs/` ЗАКОММИЧЕН** (отслеживается, публикуется на GitHub Pages через CNAME `ui.invoicebox.ru`, который туда пишет `release.sh`). Не путайте их между собой.
- **`sideEffects: false` + двойная сборка CJS/ESM — это то, что делает библиотеку tree-shakeable для потребителей.** `babel.config.json` (CJS) и `babel.config.esm.json` (ESM) отличаются только настройкой `modules`. Не добавляйте побочный эффект верхнего уровня (глобальную инъекцию CSS, создание синглтона) ни в один модуль — это молча ломает tree-shaking для каждого потребителя.
- **`src/index.ts` — это вся публичная поверхность API** (77 именованных экспортов). Компонент, не экспортированный там, не является частью пакета, независимо от того, что есть в `src/`.
- **Нигде в `src/` нет использования `.env`/`process.env`** — эта библиотека не принимает runtime-конфигурацию через окружение. Если возникает соблазн её добавить, скорее всего, это неправильный уровень (место для этого — приложение-потребитель).

## Рабочие соглашения

- Включён `strict: true` в TypeScript, и кодовая база действительно чистая под этим режимом (4 случая использования `any`, 0 non-null assertions) — сохраняйте так и дальше; не вводите новые `any`/`!`, чтобы обойти ошибку типов, находите настоящий тип.
- Нигде нет анти-паттерна clickable-div — интерактивные элементы это настоящие `<button>`. Продолжайте в том же духе.
- `Autocomplete` ([src/components/form/Autocomplete/components/Autocomplete/index.tsx](src/components/form/Autocomplete/components/Autocomplete/index.tsx)) — эталонная реализация ARIA + навигации клавиатурой в этой кодовой базе — у большинства других интерактивных компонентов (`Select`, `Dropdown`, `Checkbox`, `Toggle`) пока нет ARIA (отслеживается в аудите). Если трогаете один из них, соответствие паттерну `Autocomplete` — это ожидаемая планка, а не сверхзадача.
- Не добавляйте собственный трекер ошибок/SDK аналитики в эту библиотеку — это UI-кит, а не приложение; такое решение принадлежит потребителю. Если нужна видимость ошибок, это должен быть опциональный колбэк-проп (`onError`), который потребитель подключает сам, а не захардкоженная зависимость.
- `markdown-to-jsx` (зависимость) сейчас используется только с захардкоженным markdown в docs-сториз Storybook — никогда не подключайте его к рендерингу markdown от потребителя/пользовательского контента без явного ревью санитизации HTML.

## Не делать без предварительного согласования

- Не менять форму публичных props или экспортируемое имя ни у одного компонента — это breaking change для каждого приложения-потребителя, использующего пакет.
- Не запускать `npm run release` — он публикует в npm и пушит в `main` без гейта CI. Это дверь в одну сторону.
- Не добавлять зависимости походя — поверхность runtime-зависимостей намеренно небольшая (5 пакетов). Новая runtime-зависимость поставляется каждому потребителю.
