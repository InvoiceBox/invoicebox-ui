# Скорость и отзывчивость UI

## Примечание про область применения

Web Vitals, CLS и code splitting по роутам — свойства *страницы*, а не библиотеки компонентов — здесь нет страницы для измерения. Что реально применимо к библиотеке: корректно ли работает tree-shaking (чтобы бандл потребителя платил только за то, что импортировано), нет ли непропорционально тяжёлых отдельных компонентов, и избегает ли библиотека сама паттернов, вызывающих сдвиг макета (например, состояний загрузки без резервирования места).

## Находки

### 1. Анализатор бандла ни разу не запускался — текущие цифры размера получены через `du`, а не через реальный разбор
- **Где:** нет `source-map-explorer`/`webpack-bundle-analyzer`/аналога в devDependencies
- **Серьёзность:** low
- **Почему важно:** [01-BASELINE-METRICS.md](01-BASELINE-METRICS.md) показывает `dist/esm` на 1.3 МБ, но это сумма всех компонентов — никто не проверял реальную стоимость каждого компонента отдельно или какие компоненты типичный потребитель реально включает в бандл после tree-shaking. Без реального прогона анализатора «раздуто ли это» — просто догадка.
- **Решение:** добавить `source-map-explorer` в devDependencies, запустить его на **примере бандла потребителя** (небольшое эталонное приложение, импортирующее 3-4 реальных компонента) после сборки `dist/esm`, а не на `dist/` самой библиотеки в изоляции — это покажет реальную стоимость для потребителя.
- **Ожидаемый эффект:** реальные цифры для базовой линии вместо оценки по размеру директории; ловит случайно не-tree-shakeable импорты (например, импорт всех локалей `react-calendar`).

### 2. `AppCardLink` безусловно поставляет 4 крупных инлайн-SVG-компонента (108–177 строк каждый)
- **Где:** [src/components/common/AppCardLink/components/](../src/components/common/AppCardLink/components/) — `GooglePlayRu.tsx` (177 строк, 24 КБ после сборки), `ApkRu.tsx`, `AppGalleryRu.tsx`, `RuStoreRu.tsx`
- **Серьёзность:** low
- **Почему важно:** это иконки бейджей магазинов приложений (Google Play, RuStore, AppGallery, APK) — потребителю почти наверняка нужен только 1-2 из них на приложение, не все 4. Подтверждено в [src/components/common/AppCardLink/index.tsx:3-6](../src/components/common/AppCardLink/index.tsx): все 4 модуля иконок импортируются на уровне модуля в объект `appCardMap`, затем индексируются по prop `variant` в рантайме — бандлеры не могут статически исключить неиспользуемые ветки при runtime-обращении к объекту, поэтому каждый потребитель `AppCardLink` платит за все 4 иконки независимо от того, какой `variant` он рендерит.
- **Решение:** разбить `AppCardLink` на отдельные компоненты по варианту (`AppCardLinkGooglePlay`, `AppCardLinkApk` и т.д.), либо оставить единый компонент, но сделать каждую иконку ленивой/динамической веткой, чтобы бандлеры могли отбросить неиспользуемые. Простейшее решение при сохранении текущего API: экспортировать каждый вариант использования `<AppCardLink variant="googlePlay" />` как отдельный компонент верхнего уровня, чтобы у tree-shaking был статический импорт для исключения.
- **Ожидаемый эффект:** потребители, использующие 1 бейдж магазина приложений, не платят за остальные 3 SVG (~70 КБ несжатыми суммарно).

### 3. Нигде нет `React.lazy`/динамического `import()` — приемлемо для библиотеки компонентов, отмечено для информации
- **Где:** весь `src/` — совпадений нет
- **Серьёзность:** информационно, действий не требуется
- Корректное поведение по умолчанию: библиотека компонентов не должна единолично решать, что лениво загружается — это решение уровня приложения-потребителя (например, ленивая загрузка содержимого `Drawer`). Чинить нечего; просто подтверждаем отсутствие случайного хелпера eager-бандлинга, который мешал бы *потребителю* лениво загружать компонент, оборачивающий экспорты этой библиотеки.

## Что уже хорошо

- **Tree-shaking настроен корректно**: `sideEffects: false` в `package.json`, двойная сборка CJS/ESM через два babel-конфига, отличающихся только настройкой `modules: "commonjs"` vs `modules: false`, и `src/index.ts` использует именованные экспорты по компоненту (`export { Select } from './components/form/Select'`), а не объект-неймспейс/дефолтный барель. Это самый важный рычаг производительности для библиотеки компонентов, и он реализован правильно.
- Компонент `Skeleton` существует и используется в 5 местах — правильный примитив для избежания сдвига макета во время загрузки, вместо того чтобы каждый потребитель изобретал своё.
- Нет тяжёлой сторонней зависимости уровня session-replay/аналитики/капчи в критическом пути вообще (подтверждено отсутствие в [09-observability.md](09-observability.md)) — ничто здесь молча не раздувает бандл каждого потребителя посторонним инструментарием.
