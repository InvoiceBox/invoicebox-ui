# Безопасность

## Примечание про область применения

CSP-заголовки, состояние OAuth/CSRF и политика хранения PII в браузере — зона ответственности **приложения-потребителя**, а не UI-кита — у библиотеки нет собственного флоу авторизации и она сама ничего не хранит (подтверждено: нигде в `src/` нет `localStorage`/`sessionStorage`/`document.cookie`). Что применимо к библиотеке компонентов: поверхность XSS в компонентах, рендерящих произвольный контент, гигиена атрибутов ссылок, CVE в зависимостях, отсутствие захардкоженных секретов.

## Находки

### 1. `AppCardLink` открывает `target="_blank"` без `rel="noopener noreferrer"`
- **Где:** [src/components/common/AppCardLink/index.tsx:26](../src/components/common/AppCardLink/index.tsx) — `<S.Wrapper href={href} target={'_blank'}>`
- **Серьёзность:** high
- **Почему важно:** классический вектор reverse-tabnabbing — страница, открытая через `target="_blank"`, получает живую ссылку `window.opener` обратно на приложение invoicebox и может незаметно перенаправить эту вкладку на фишинговый двойник, пока пользователь смотрит в новую вкладку. Это единственный `target="_blank"` во всей кодовой базе, и это единственный случай без митигации.
- **Решение:** добавить `rel="noopener noreferrer"` в styled-компонент ссылки (правка в одну строку в styled-components `S.Wrapper` либо в JSX).
- **Ожидаемый эффект:** закрывает вектор reverse-tabnabbing; никаких изменений поведения для легитимного использования.

### 2. `markdown-to-jsx` рендерит сырой markdown в docs-сториз Storybook — стоит подстраховаться до реального использования
- **Где:** 4 файла `docs.stories.tsx` (`DateInput`, `Calendar`, `Toast`, `Typography`) — все рендерят **захардкоженные** строки markdown, ничего из этого не экспортируется в `src/index.ts`, то есть потребителю это не поставляется
- **Серьёзность:** low (сейчас риска нет — отмечено, чтобы не допустить превращения этого в публичный экспорт без ревью)
- **Почему важно:** `markdown-to-jsx` может рендерить HTML, встроенный в markdown, в зависимости от конфигурации; сейчас вход полностью статичный/захардкоженный, поэтому пути инъекции нет. Если будущий компонент превратит это в реальную фичу «рендерить markdown из prop», этот prop станет недоверенным вводом, и потребуется явно ограничить опции рендеринга HTML в библиотеке (или добавить санитайзер) до релиза.
- **Решение:** изменений кода сейчас не требуется. Добавить короткую заметку рядом с зависимостью (или в `CLAUDE.md`) о том, что любой компонент на базе `markdown-to-jsx` не должен рендерить markdown, полученный от потребителя, без `options={{ disableParsingRawHTML: true }}` (или аналога), с ревью перед мёржем.
- **Ожидаемый эффект:** предотвращает регресс XSS от будущей благонамеренной фичи.

### 3. Нигде нет `dangerouslySetInnerHTML`
- **Где:** весь `src/` — проверено грепом, совпадений нет
- **Серьёзность:** отсутствует (зафиксировано для полноты картины / как базовая линия)
- Это основной вектор XSS в React, и он просто не используется. Сохраняйте так и дальше — если будущему компоненту понадобится рендерить rich/HTML-контент, это должно идти через санитайзер (DOMPurify или аналог), а не через сырой `dangerouslySetInnerHTML`, и это решение должно получить отдельное ревью.

### 4. Нет слоя валидации URL у компонентов, принимающих `href` — по замыслу, но не задокументировано
- **Где:** [src/components/common/AppCardLink/index.tsx](../src/components/common/AppCardLink/index.tsx) принимает `href: string` и рендерит его напрямую; `SecondaryButton` аналогично может рендериться как ссылка
- **Серьёзность:** low
- **Почему важно:** это не баг библиотеки — UI-кит, рендерящий любой переданный `href`, это корректная архитектурная граница (библиотека не должна владеть политикой навигации/модели угроз, которая принадлежит приложению-потребителю). Но сейчас это *неявный* контракт. Если потребитель передаёт несанитизированную пользовательскую строку прямо в `href` (например, `javascript:`-URI), это эксплуатируется в его приложении, и ничто здесь его не предупреждает.
- **Решение:** явно задокументировать (в README или JSDoc на проп), что props ссылок/навигации рендерятся как есть, и потребитель отвечает за валидацию URL перед передачей — в соответствии с принципом «валидировать на границах системы», просто с явным указанием, чья это граница.
- **Ожидаемый эффект:** убирает неоднозначность для потребителей, интегрирующих библиотеку в приложения с реальной моделью угроз (например, с пользовательским контентом).

### 5. Нет захардкоженных секретов, API-ключей или дефолтов, похожих на учётные данные
- **Где:** весь `src/` — грепнуто по паттернам `api[_-]?key|secret|token|password`; единственное совпадение — `autoComplete={'new-password'}` в `PhoneInput`, легитимная HTML-подсказка автозаполнения, не реальный секрет
- **Серьёзность:** отсутствует — зафиксировано как чистая базовая линия

### 6. `npm audit`: 85 известных уязвимостей, 3 критических — все в дереве devDependency/build-tool
- **Где:** `package.json` devDependencies, в основном цепочка Storybook 7 / `@storybook/preset-create-react-app` / webpack, плюс `@babel/core`
- **Серьёзность:** medium (не high, потому что ничего из этого не поставляется потребителям — `files: ["dist"]` означает, что публикуется только собранный из `src/` вывод, и 5 runtime-зависимостей тут не при чём) — но всё равно стоит починить, так как это влияет на машины контрибьюторов и на CI (когда он появится), и 3 уязвимости имеют рейтинг critical (`form-data`, `handlebars`, `shell-quote`)
- **Решение:** см. [07-platform.md](07-platform.md) — план обновления зависимостей — по сути это проблема «Storybook 7 отстаёт на 3 мажора», лучше решается мажорным апгрейдом Storybook, чем точечными `npm audit fix --force`, которые конфликтуют с lock-файлом.
- **Ожидаемый эффект:** устраняет 3 critical / 31 high находку; снижает шум `npm audit` для всех, кто его запускает в будущем.

## Что уже хорошо

- Ноль `dangerouslySetInnerHTML`, ноль захардкоженных секретов, ноль использования клиентского хранилища (`localStorage`/`sessionStorage`/`cookie`) — действительно чистая базовая линия по самым серьёзным векторам XSS/утечки секретов.
- Поверхность runtime-зависимостей намеренно небольшая (5 пакетов), и ни одна из них не фигурирует в находках `npm audit` — вся CVE-экспозиция полностью сосредоточена в dev/build-инструментарии, что и есть правильное место для неё, если ей вообще где-то быть.
- `sideEffects: false` + отсутствие глобальной мутации `window`/`document` за пределами portal `Drawer` (`document.body.style.overflow`, корректно очищается при размонтировании) — нет признаков утечки глобального состояния между экземплярами компонента.
