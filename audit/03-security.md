# 03 — Безопасность

Модель угроз для UI-kit: компонент получает данные/URL от приложения-потребителя,
часть которых может происходить из пользовательского ввода. Библиотека обязана не
создавать каналов XSS/open-redirect и не тянуть уязвимые зависимости в прод.

Хорошие новости: `dangerouslySetInnerHTML` = 0, секретов/ключей/токенов в коде = 0
(единственное совпадение — `autoComplete="new-password"`, легитимно). Сети/OAuth/CSRF
в библиотеке нет — эти пункты ТЗ неприменимы.

---

### S-1. `AppCardLink`: непровалидированный `href` + `target="_blank"` без `rel`
- **Где:** `src/components/common/AppCardLink/index.tsx:26` — `<S.Wrapper href={href} target={'_blank'}>`.
- **Серьёзность:** **high**.
- **Описание:** два дефекта в одной строке:
  1. `href` рендерится «как есть». Если потребитель передаст `javascript:…` (или
     `data:`) URL из пользовательского ввода — это XSS при клике (DOM-based).
  2. `target="_blank"` без `rel="noopener noreferrer"` — reverse tabnabbing и утечка
     `Referer`. (Современные браузеры по умолчанию ставят noopener для `_blank`, но
     полагаться на это нельзя — это поведение и старые webview.)
- **Решение:**
  - Валидировать схему: разрешать только `https:`/`http:` (и относительные), отклонять
    `javascript:`/`data:`/`vbscript:`. Реализовать `isSafeUrl(url)` в `utils`.
  - Жёстко проставлять `rel="noopener noreferrer"` для всех `target="_blank"`.
- **Эффект:** закрыт DOM-XSS и tabnabbing на самой частой ссылочной точке библиотеки.
- **Best practice:** OWASP DOM XSS Prevention; React docs — «only `http`/`https` URLs».

### S-2. `Validate.website` — небезопасный для повторного использования как «санитайзер»
- **Где:** `src/validate/index.ts` (метод `website`, экспортируется в публичный API).
- **Серьёзность:** medium.
- **Описание:**
  1. Регулярка **не заякорена справа** (`^(…)…?` без `$`) — валидирует «начинается с»,
     а не «является» URL. Для проверки формата поля терпимо, но опасно, если значение
     потом используется как `href`.
  2. Схема опциональна (`(?:https?:\/\/)?`) — `abc.com` проходит без протокола; при
     построении ссылки потребитель обязан сам добавить `https://` (иначе относительный
     путь → open-redirect внутри своего домена).
  3. Вложенные квантификаторы (`{0,61}…)+ … )*`) — потенциальный **ReDoS** на
     спец-входе.
- **Решение:** добавить правый якорь `$`; в JSDoc явно указать «валидация формата, НЕ
  санитизация для href»; заменить на безопасный парсинг через `new URL()` где возможно;
  прогнать regex через ReDoS-чек (например, `recheck`/`safe-regex`).
- **Эффект:** валидатор перестаёт быть источником ложной уверенности и ReDoS-вектором.

### S-3. 83 уязвимости в зависимостях (3 critical, 31 high)
- **Где:** `npm audit` — преимущественно dev-цепочка Storybook 7 / `react-scripts` /
  `@svgr/*` / `workbox-*` / `rollup-plugin-terser`.
- **Серьёзность:** critical (формально), фактический риск — в основном dev-time.
- **Описание:** critical: `form-data`, `handlebars`, `shell-quote`. Большинство — не
  попадает в опубликованный `dist` (только peer `react`/`styled-components` идут в прод),
  но влияет на безопасность сборочной машины и доверие к пакету.
- **Решение:** см. `07-platform.md` (апгрейд Storybook 8/9, отказ от `react-scripts`/
  `@storybook/preset-create-react-app`, который и тянет `workbox`/`svgr`). Зафиксировать
  `npm audit` в CI с порогом.
- **Эффект:** снижение CVE-поверхности на десятки пунктов, чистый `audit` как гейт.

### S-4. CSP / защитные заголовки — для Storybook на GitHub Pages
- **Где:** публикация `docs/` (Storybook) на `ui.invoicebox.ru`.
- **Серьёзность:** low.
- **Описание:** статическая витрина без CSP/`X-Content-Type-Options`/`Referrer-Policy`.
  Для самой npm-библиотеки CSP неприменим (заголовки задаёт приложение-потребитель).
- **Решение:** добавить мета-CSP в `.storybook/preview-head.html` или заголовки на
  уровне хостинга; в README библиотеки — рекомендации по CSP для потребителей.
- **Эффект:** базовая защита витрины, явный гайд для интеграторов.

### S-5. `hex-to-rgba` и ручные строковые операции с цветом
- **Где:** 17 файлов используют `hex-to-rgba`; `utils/transformColorByHsl.ts`.
- **Серьёзность:** low.
- **Описание:** цвет приходит из палитры потребителя; некорректный ввод не
  валидируется — не XSS (вставляется в CSS через styled-components, который
  экранирует), но возможен «битый» стиль.
- **Решение:** валидация/нормализация цвета на входе в `PaletteProvider`.
- **Эффект:** устойчивость темизации к некорректным значениям.
