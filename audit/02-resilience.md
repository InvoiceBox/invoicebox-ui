# 02 — Отказоустойчивость (обработка ошибок)

Принцип для UI-kit: компонент не должен «ронять» дерево приложения-потребителя.
Сейчас обработка ошибок отсутствует как класс — `try/catch` = 0, ErrorBoundary = 0.

---

### R-1. Нет ни одного ErrorBoundary в библиотеке
- **Где:** весь `src` (0 совпадений `componentDidCatch`/`getDerivedStateFromError`).
- **Серьёзность:** medium (для библиотеки — спорно: ErrorBoundary чаще ответственность
  приложения, но «составные» компоненты с асинхронным контентом стоит защищать).
- **Описание:** если внутренний рендер падает (например, кривые данные в `Autocomplete`/
  `Select`/`Calendar`), исключение всплывает в приложение-потребитель и валит весь экран.
- **Решение:** экспортировать опциональный `<ErrorBoundary>` и оборачивать им
  «тяжёлые» составные компоненты (Autocomplete, DateTimeInput, Calendar) изнутри
  с безопасным fallback. Не навязывать — давать проп `renderError`.
- **Эффект:** локализация сбоя одним компонентом вместо падения всей страницы.

### R-2. Парсинг/числовые операции без защиты от мусора
- **Где:** `src/validate/index.ts` (`parseInt(value.slice…)`, `Number(bikRs[i])`),
  `form/CurrencyInput/*`, `form/PositiveIntegerInput`, `form/DateInput`/`DateTimeInput`.
- **Серьёзность:** medium.
- **Описание:** `parseInt`/`Number` на неожиданном вводе дают `NaN`, который тихо
  протекает в контрольные суммы и расчёты дат — невалидный результат без явной ошибки.
- **Решение:** guard-проверки (`Number.isFinite`), ранний возврат `createError`,
  нормализация ввода до вычислений. Покрыть тестами edge-cases.
- **Эффект:** предсказуемые ошибки валидации вместо «тихого» NaN.

### R-3. `useEffect` с таймерами/подписками — проверить очистку
- **Где:** `common/Drawer/index.tsx` (setTimeout 300мс — очистка есть ✅),
  `hooks/useOutsideClick`, `hooks/useMedia`, компоненты с `addEventListener`.
- **Серьёзность:** low.
- **Описание:** часть эффектов корректна; нужен сплошной аудит на утечки слушателей/
  таймеров при размонтировании.
- **Решение:** убедиться, что каждый `addEventListener`/`setTimeout`/`setInterval`
  имеет cleanup; добавить ESLint `react-hooks/exhaustive-deps` (сейчас только recommended).
- **Эффект:** отсутствие утечек памяти и «фантомных» обработчиков.

### R-4. Асинхронные обработчики без `.catch`
- **Где:** `hooks/useLoadingSubmit`, компоненты с `onChange`/`onSubmit`, прокидывающие
  промисы потребителя.
- **Серьёзность:** medium.
- **Описание:** линт уже включает `no-floating-promises`/`no-misused-promises` (хорошо),
  но в рантайме отклонённый промис из колбэка потребителя может оставить спиннер навсегда.
- **Решение:** в `useLoadingSubmit` оборачивать вызов в `try/finally`, гарантируя сброс
  `isLoading` даже при reject; задокументировать контракт.
- **Эффект:** UI не «залипает» в состоянии загрузки при ошибке сабмита.

### R-5. Нет деградации при отсутствии PaletteProvider
- **Где:** `palette/useComponentPalette.ts`, `usePalette.ts`.
- **Серьёзность:** low/medium.
- **Описание:** если потребитель забыл обернуть в `PaletteProvider`, доступ к палитре
  может бросить/вернуть undefined → падение стилей.
- **Решение:** дефолтная палитра как fallback + понятный `console.warn` (dev-only).
- **Эффект:** компоненты рендерятся с дефолтной темой, а не падают.

---

**Приоритет блока:** R-2 и R-4 (medium) — в волну 1; R-1/R-5 — волна 2; R-3 — фоном.
