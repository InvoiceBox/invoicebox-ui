# Отказоустойчивость и обработка ошибок

## Примечание про область применения

Эта библиотека не делает **ни одного** сетевого запроса сама (проверено: нигде в `src/` нет `fetch`/`axios`/`XMLHttpRequest`/`.then`/`async function`). Она полностью презентационная — данные и колбэки приходят от потребителя через props. Это значит, что большая часть чек-листа отказоустойчивости из ТЗ (таймауты, ретраи только GET с бэкоффом, офлайн-режим, 429/`Retry-After`, поллинг статусов) — **зона ответственности приложения-потребителя, а не библиотеки** — здесь просто нечего ретраить и нечего таймаутить. Что реально применимо: error boundary вокруг рискованных поддеревьев, «молчаливые» catch-и и коалесинг «бури» тостов.

## Находки

### 1. Нигде в библиотеке нет React Error Boundary
- **Где:** весь `src/` (грепнуто на `ErrorBoundary`, `componentDidCatch`, `getDerivedStateFromError` — совпадений нет)
- **Серьёзность:** medium
- **Почему важно:** компоненты вроде `Drawer`, `Dropdown`, `Autocomplete`, `Select` рендерят детей/render-prop контент, предоставленный потребителем (`renderOption`, `renderGroup`, `renderValue`, `children`). Если пользовательская render-логика бросает исключение, размонтируется всё React-дерево, в котором находится компонент библиотеки — без границы, которая бы ограничила последствия одним виджетом.
- **Решение:** обернуть компоненты, рендерящие пользовательский контент (`Select`, `Autocomplete`, `Drawer`), в локальный error boundary с деградацией до минимального безопасного состояния вместо краха.
- **Ожидаемый эффект:** сломанный колбэк `renderOption` ломает один дропдаун, а не всю страницу потребителя.

### 2. Ноль блоков `try`/`catch` во всём кодовой базе
- **Где:** весь `src/`
- **Серьёзность:** low
- **Почему важно:** само по себе не плохо — асинхронного I/O здесь нет. Но это также значит, что нет защитной обработки вокруг парсинга пользовательского ввода, который *теоретически* может бросить исключение (например, некорректные даты/номера телефонов, переданные как начальные значения). Отмечено как low, а не пропущено, поскольку «нет try/catch» звучит иначе, когда появится error boundary (пункт 1) в качестве страховочной сетки.
- **Решение:** действий не требуется, пока не решён пункт 1 — тогда пересмотреть, нужны ли конкретным путям парсинга (форматирование даты/телефона/валюты) локальные защиты вместо полного полагания на boundary.
- **Ожидаемый эффект:** N/A до объединения с пунктом 1.

### 3. У Toast нет дедупликации/коалесинга
- **Где:** [src/components/common/Toast/index.tsx](../src/components/common/Toast/index.tsx) — тонкая обёртка над `toast.error`/`toast.success` из `react-toastify`, без ключа дедупликации, без ограничения частоты
- **Серьёзность:** low
- **Почему важно:** если приложение-потребитель вызывает один и тот же тост об ошибке в цикле (например, повторяющийся запрос, каждый раз падающий одинаково), ничто здесь не мешает стеку одинаковых тостов — «буря тостов», как названо в ТЗ. Это обёртка-удобство самой библиотеки, так что чинить логично здесь, а не заставлять каждого потребителя реализовывать дедупликацию заново.
- **Решение:** добавить опциональный `toastId`, производный от сообщения (`react-toastify` нативно поддерживает `toastId` именно для этого — `toast.error(content, { toastId })`), чтобы одинаковые сообщения схлопывались, а не стекались.
- **Ожидаемый эффект:** повторяющиеся одинаковые ошибки показываются один раз, а не N раз.

### 4. Использование `setTimeout` — всё про UI-тайминги, ничего про сеть
- **Где:** `PhoneInput/phoneInputLogic.ts:127` (позиционирование каретки), `TagsInput/index.tsx` (тайминг автозаполнения), `DateTimeInput/components/MobileNumbersColumn/index.tsx` (центрирование скролла), `DateTimeInput/hooks/useTimePickerHeight.ts` (измерение высоты), `Drawer/index.tsx` (таймер 3000мс на завершение анимации перед размонтированием)
- **Серьёзность:** информационно, действий не требуется
- Подтверждает, что нигде не спрятана логика ретраев/бэкоффа, которую нужно было бы проверять по правилу «ретраить только идемпотентный GET» — такой логики просто нет, ни сетевой, ни какой-либо другой.

## Что уже хорошо

- Корректно **ноль** сетевого кода — нет риска, что компонент случайно заретраит неидемпотентный POST, потому что ретраить нечего. Здесь чинить нечего; сохраняйте так и дальше при добавлении новых компонентов (рискованным станет новый компонент, начавший делать вызовы сам, а не делегирующий их через props/колбэки).
- Размонтирование `Drawer` использует реальный таймер завершения анимации (на основе `ANIMATION_MS`), а не произвольную догадку — избегает распространённого бага с размонтированием посреди перехода.
