# Тесты

## Находки

### 1. По сути 1 тестовый файл на всю библиотеку
- **Где:** [src/validate/utils/websiteValidate.test.ts](../src/validate/utils/websiteValidate.test.ts) — единственный реальный тест в кодовой базе (6 тест-кейсов, покрывает логику валидации URL). ~41 компонент и 6 хуков не имеют тестов вообще.
- **Серьёзность:** critical
- **Почему важно:** это общая библиотека UI, используемая другими приложениями — регресс в `Select`, `DateTimeInput` или логике форматирования телефона/валюты молча ломает каждое приложение-потребитель, обновившее версию, и ничто это не поймает до публикации (усугубляется отсутствием CI из [07-platform.md](07-platform.md)).
- **Решение:** начать с двух самых ценных и наименее затратных категорий согласно приоритету из ТЗ: (1) юнит-тесты чистой логики — `validate/` (логика ИНН/VAT-номеров рядом с уже существующим валидатором сайтов), `utils/hexToRgba`, `utils/transformColorByHsl`, и логика форматирования внутри `phoneInputLogic.ts`; (2) тесты на уровне компонентов для самых сложных/рискованных компонентов — `Select`, `Autocomplete`, `DateTimeInput` — с использованием React Testing Library (пока не зависимость — см. находку 3) для проверки взаимодействия с клавиатурой, выбора опций и ветвления вариантов single/multi-select.
- **Ожидаемый эффект:** ловит регрессии логики в компонентах, где они наиболее вероятны (самые крупные, самые стейтфул), до попадания в npm.

### 2. Jest запускается вообще без конфигурации — из-за этого один и тот же тест выполняется 3× локально
- **Где:** нет файла `jest.config.*`, нет ключа `"jest"` в [package.json](../package.json)
- **Серьёзность:** high
- **Почему важно:** подтверждено вживую — `npm test` показывает «3 test suites», хотя реально существует 1 уникальный тестовый файл; остальные 2 — тот же тест, подхваченный из локально собранных `dist/` и `dist/esm/` (в gitignore, поэтому на чистый чекаут CI без предварительной сборки это не влияет, но это реальная ловушка для локальной отладки, и станет реальной проблемой, как только появится CI, запускающий `build` перед `test` в одной джобе).
- **Решение:** добавить `jest.config.js` (или ключ `"jest"` в `package.json`) с `testPathIgnorePatterns: ["<rootDir>/dist/", "<rootDir>/node_modules/"]`.
- **Ожидаемый эффект:** `npm test` показывает реальное количество test suites; безопасно запускать `test` после `build` в одной CI-джобе без двойного счёта.

### 3. Вообще не установлена библиотека для тестирования компонентов
- **Где:** devDependencies в `package.json` — нет `@testing-library/react`, тест-раннер Storybook не подключён к запуску в CI (аддон `@storybook/test` присутствует, но это для интерактивных тестов *внутри* сториз Storybook, а не отдельного пути `npm test`)
- **Серьёзность:** high
- **Почему важно:** даже когда появятся тестовые файлы (находка 1), нет инструментов для рендеринга компонента и проверки его DOM-вывода/взаимодействий — тесты чистых функций сами по себе не могут покрыть навигацию клавиатурой в `Select` или поведение фокуса в `Drawer`.
- **Решение:** добавить `@testing-library/react` + `@testing-library/user-event` в devDependencies (они совместятся с уже существующим Jest 29 без конфликтов конфигурации). Дополнительно/альтернативно — подключить тест-раннер Storybook (`@storybook/test-runner`) для запуска существующих и будущих интерактивных сториз в CI — уже присутствующая зависимость `@storybook/test` наводит на мысль, что таков был изначальный замысел, просто он никогда не был подключён к реальному запуску в CI.
- **Ожидаемый эффект:** делает тестирование на уровне компонентов (находка 1) реально возможным, а не только теоретически желаемым.

### 4. Тесты не запускаются как гейт CI — потому что CI не существует
- **Где:** см. находку 1 в [07-platform.md](07-platform.md)
- **Серьёзность:** critical (та же первопричина, отмечена здесь ради специфичных для тестов последствий)
- **Почему важно:** даже хороший набор тестов не даёт никакой защиты, если ничто не заставляет его проходить перед тем, как `release.sh` сможет опубликовать пакет.
- **Решение:** то же решение, что и в находке 1 документа про платформу — порядок стадий CI должен быть lint → typecheck → test → build, тесты как жёсткий гейт, до того, как у специфичной для этой библиотеки работы по написанию тестов (находки 1-3) появится хоть какая-то сила.
- **Ожидаемый эффект:** тесты реально блокируют плохие релизы, а не остаются необязательной документацией намерений.

### 5. Нет настройки e2e/интеграционных тестов
- **Где:** нигде нет конфига или зависимости Cypress/Playwright
- **Серьёзность:** low
- **Почему важно:** отмечено для полноты по чек-листу из ТЗ, но для библиотеки компонентов (не приложения с критическим пользовательским путём/чекаутом для защиты end-to-end) полноценный e2e ниже по приоритету, чем тесты на уровне компонентов (находка 3) — здесь нет единого «критического пути», аналогичного чекауту, который оправдывал бы инвестиции в Cypress/Playwright прямо сейчас.
- **Решение:** отложить. Интерактивные тесты Storybook (второй вариант из находки 3) покрывают большую часть того, что дал бы e2e, при значительно меньшей стоимости настройки/поддержки, учитывая природу проекта.
- **Ожидаемый эффект:** избегает избыточных инвестиций в e2e-инструментарий, не соответствующий форме этого проекта.

## Что уже хорошо

- Единственный существующий тест (`websiteValidate.test.ts`) — хороший пример того, как должно выглядеть тестирование чистой логики здесь: `describe`/`test`/`expect`, понятные названия кейсов («validate web sites all variants», «validate web sites only https variants»), проверка реальных граничных случаев (интернационализированные домены, пути, query-параметры, режим только https), а не только тривиальный happy-path. Используйте его как шаблон для валидатора VAT-номеров и других тестов чистой логики из находки 1.
- Jest 29 (текущая, не сильно устаревшая версия) уже выбран как раннер, `@types/jest` присутствует — миграция инструментария не нужна, чтобы начать писать больше тестов, нужна только конфигурация (находка 2) и зависимость для тестирования компонентов (находка 3).
