## Шкала серьёзности

- **critical** — реально сломано или эксплуатируется, чинить в первую очередь
- **high** — реальное влияние на пользователя/потребителя пакета, обходного пути нет
- **medium** — реальное влияние есть, но есть обходной путь, либо затрагивает меньшинство компонентов
- **low** — гигиена/best-practice, известных инцидентов нет

## Что важно держать в голове по ходу всего документа

`@invoicebox/ui` — это **опубликованная npm-библиотека компонентов**, а не приложение. У неё нет роутов, нет глобального стора, нет SSR, нет собственных сетевых запросов и нет страниц, на которых можно измерить Lighthouse. Часть категорий из исходного технического задания (офлайн-режим, поллинг статусов, CSP-заголовки, OAuth/CSRF) **не применима** здесь по замыслу — это зона ответственности приложения-потребителя, а не библиотеки. В каждом документе явно указано, где категория N/A и почему, а не просто пропущена молча.
