# Доступность (WCAG 2.1 AA)

## Находки

### 1. Только 2 из ~41 компонента имеют хоть какие-то ARIA/role-атрибуты
- **Где:** весь `src/components/` — `Autocomplete` (`role="combobox"`/`"listbox"`/`"option"`, `aria-expanded`, `aria-controls`, `aria-autocomplete`, `aria-activedescendant`, `aria-selected`) и `Drawer` (`role="dialog"`, `aria-modal="true"`) — единственные компоненты с ARIA. У остальных ~39 — включая `Select`, `Dropdown`, `Tooltip`, `Checkbox`, `Toggle`, `TagsInput`, `CountrySelect`, `PhoneInput`, `DateInput`, `DateTimeInput`, `Calendar` — ничего нет.
- **Серьёзность:** high
- **Почему важно:** пользователи скринридеров не получают семантической информации для большинства интерактивных виджетов кита — `Toggle` или `Checkbox` без role/state непригодны для использования со вспомогательными технологиями, а поскольку это библиотека *компонентов*, разрыв тиражируется в каждое приложение-потребитель, использующее её.
- **Решение:** приоритизировать интерактивные формовые компоненты в первую очередь (`Checkbox`, `Toggle`, `Select`, `Dropdown`) — добавить `role`, `aria-checked`/`aria-selected`/`aria-expanded` где уместно, следуя паттерну, уже корректно реализованному в `Autocomplete`. Использовать его как внутренний эталон.
- **Ожидаемый эффект:** пользователи скринридеров реально смогут работать с формовыми компонентами, составляющими основную часть использования кита.

### 2. У `Drawer` есть `role="dialog"` + `aria-modal`, но нет фокус-трапа и нет возврата фокуса при закрытии
- **Где:** [src/components/common/Drawer/index.tsx](../src/components/common/Drawer/index.tsx) — подтверждено: нет `focus-trap-react`, нет ручной логики захвата клавиатуры через `useRef`/`useEffect`, нет кода, сохраняющего `document.activeElement` перед открытием и восстанавливающего его при закрытии
- **Серьёзность:** high
- **Почему важно:** модалка без фокус-трапа позволяет пользователям клавиатуры/скринридера табом выйти на страницу позади неё — это перечёркивает саму цель `aria-modal="true"`, которая обещает вспомогательным технологиям, что фокус изолирован. Отсутствие возврата фокуса при закрытии означает, что пользователь теряет своё место после закрытия.
- **Решение:** добавить фокус-трап (небольшая зависимость вроде `focus-trap-react`, либо ручная циклическая обработка `Tab`/`Shift+Tab` внутри DOM-поддерева drawer) и захват/восстановление фокуса элемента-триггера при открытии/закрытии.
- **Ожидаемый эффект:** соответствует паттерну модального диалога WCAG 2.1 AA; соответствует тому, что уже обещают `role="dialog"`/`aria-modal`.

### 3. У `Dropdown` и `Tooltip` вообще нет семантики диалога или управления фокусом
- **Где:** [src/components/common/Dropdown/index.tsx](../src/components/common/Dropdown/index.tsx), [src/components/common/Tooltip/index.tsx](../src/components/common/Tooltip/index.tsx)
- **Серьёзность:** medium
- **Почему важно:** `Dropdown` используется как механизм всплывающего окна внутри `Select`, поэтому этот разрыв распространяется на один из самых используемых формовых компонентов. Нет role, нет обработки фокуса — у пользователей, работающих только с клавиатурой, нет гарантии связного порядка табуляции в/из содержимого поповера.
- **Решение:** для `Dropdown` — добавить `role="menu"`/`"listbox"` в зависимости от контекста использования, и обеспечить участие в той же обработке фокус-трапа, что и `Drawer`, когда используется как модалоподобный поповер. Для `Tooltip` — добавить `role="tooltip"` и связать его с элементом-триггером через `aria-describedby`, а не полагаться исключительно на визуальное состояние hover/focus.
- **Ожидаемый эффект:** паритет клавиатуры и скринридера для самого распространённого паттерна взаимодействия (`Select` → `Dropdown`).

### 4. Нигде нет поддержки `prefers-reduced-motion`, несмотря на несколько анимированных компонентов
- **Где:** весь `src/` — `Drawer` (переход 300мс), `InputLabel` (keyframes `labelFadeIn`), различные правила `transition: all 0.2s ease-in-out` в styled-components — ни одной защиты через `@media (prefers-reduced-motion: reduce)` не найдено
- **Серьёзность:** medium
- **Почему важно:** пользователи с вестибулярными нарушениями, установившие системную настройку reduced-motion, получают те же анимации, что и все остальные — прямой разрыв с WCAG 2.3.3 (AAA, но хорошая практика и для китов, ориентированных на AA) и общий пробел в лучших практиках доступности.
- **Решение:** добавить общий styled-components миксин `prefers-reduced-motion` (проверяется один раз в общих утилитах темы/`palette`) и применить его везде, где есть правило `transition`/`keyframes`, переключаясь на `transition: none` / мгновенные изменения состояния, когда медиазапрос совпадает.
- **Ожидаемый эффект:** соблюдение системной настройки доступности во всём ките одним общим миксином, а не 10 точечными правками.

### 5. Не установлен плагин ESLint `jsx-a11y`
- **Где:** [.eslintrc.json](../.eslintrc.json) — массив `plugins` содержит `react`, `@typescript-eslint`, `prettier`, `react-hooks`, `jest`; `jsx-a11y` отсутствует
- **Серьёзность:** medium
- **Почему важно:** без `eslint-plugin-jsx-a11y` ни одна из находок 1-4 выше не отлавливалась бы автоматически в будущем — каждый новый компонент может повторно допустить те же пробелы без сигнала линтера.
- **Решение:** добавить `eslint-plugin-jsx-a11y`, подключить `plugin:jsx-a11y/recommended`, выставить `error` (не `warn`) — в соответствии со сквозной темой аудита «правила линтера как страховка от регрессий» (см. [06-architecture-lint.md](06-architecture-lint.md)).
- **Ожидаемый эффект:** новые регрессии доступности отлавливаются на этапе PR, а не обнаруживаются пользователями.

### 6. Тач-таргеты: смешанная картина — некоторые компоненты соответствуют 44×44, у нескольких, вероятно, нет
- **Где:** `CodeInput` (высота 48–54px), `Calendar` (48px), `Toast` (min-height 54px) соответствуют рекомендации. Меньшие интерактивные элементы (чекбоксы, чипы, спиннеры) не проверены на ≥44px — реальный размер на экране зависит от props отступов, которые варьируются в зависимости от использования, а не захардкожены в стилях, поэтому нужен рантайм/визуальный замер, а не статический грep.
- **Серьёзность:** medium
- **Решение:** проверить реальный отрендеренный размер `Checkbox`, `Toggle`, `Chip` и `SecondaryButton` в дефолтной/наиболее распространённой конфигурации props через инспектор браузера (не грep — размер здесь задаётся props, а не захардкожен), и установить задокументированный минимальный дефолтный отступ там, где что-то оказывается меньше 44×44.
- **Ожидаемый эффект:** согласованный размер тач-таргетов по дефолтным конфигурациям кита; потребители, явно переопределяющие размер, сами отвечают за свой выбор.

## Что уже хорошо

- `Autocomplete` — действительно солидная эталонная реализация: полный набор ARIA (`combobox`/`listbox`/`option` роли, `aria-expanded`/`aria-controls`/`aria-autocomplete`/`aria-activedescendant`/`aria-selected`) плюс реальная навигация клавиатурой (обработчики ArrowUp/ArrowDown/Enter/Escape). Используйте его как внутренний паттерн для тиражирования в `Select`/`Dropdown`/`Checkbox`/`Toggle`, а не изобретайте новый подход.
- **Нигде нет анти-паттерна clickable-`<div>`/`<span>`** — каждый найденный интерактивный элемент использует настоящий `<button>` либо делегирует через семантический элемент. Это распространённый провал доступности в других проектах, и здесь его уже избежали.
- Использование `tabIndex={-1}` (3 случая) применено корректно — убирает уже фокусируемые элементы из порядка табуляции для программно управляемых пунктов (например, обёртки опций в `Autocomplete`), а не распространённое злоупотребление принудительным порядком через положительные значения.
- `inputMode`/`autoComplete` уже подключены там, где нужны больше всего: `PhoneInput` (`inputMode="numeric"`), `CodeInput` (`inputMode="numeric"`, `autoComplete="one-time-code"`), `TextInput` (оба пропа прокидываются универсально).
- Нигде не найдено ограничение `user-scalable=no`/`maximum-scale` — зум не заблокирован.
