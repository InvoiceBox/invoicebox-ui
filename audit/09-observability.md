# 09 — Наблюдаемость

> В ТЗ пункт 8 оставлен без расшифровки. Для **библиотеки** наблюдаемость трактуем как:
> предсказуемое логирование, диагностируемость в потребителе, версионирование/changelog,
> и хуки для интеграции с мониторингом приложения-потребителя (а не встроенный SDK
> телеметрии — это было бы навязыванием).

---

### O-1. Бессистемное логирование (`console.*`)
- **Где:** 3 вхождения `console.*`, в т.ч. `validate/index.ts`
  (`console.error('Form dont have vatNumber or countryId')`), линт-правило `no-console: warn`.
- **Серьёзность:** medium.
- **Описание:** библиотека пишет напрямую в `console` потребителя, без уровня/префикса/
  возможности отключить. Шум в проде, нет контроля.
- **Решение:** единый dev-only логгер с префиксом `[invoicebox/ui]`, отключаемый в
  production-сборке (`process.env.NODE_ENV`), без прямых `console` в рантайм-коде.
- **Эффект:** чистый прод-лог потребителя, управляемая диагностика.

### O-2. Нет механизма репортинга ошибок наружу
- **Где:** компоненты с асинхронной логикой / валидацией.
- **Серьёзность:** medium.
- **Описание:** когда внутри компонента что-то идёт не так (см. `02-resilience`), у
  потребителя нет колбэка/хука узнать об этом и отправить в свой Sentry/мониторинг.
- **Решение:** опциональный проп `onError`/контекст `onError` на составных компонентах
  и в ErrorBoundary (R-1); документировать.
- **Эффект:** потребитель видит сбои библиотеки в своём мониторинге.

### O-3. Нет CHANGELOG / прозрачности версий
- **Где:** `release.sh` бампит `prerelease --preid=alpha` без changelog; `RELEASE.md`
  описывает процесс, но истории изменений API нет.
- **Серьёзность:** low/medium.
- **Описание:** потребитель не понимает, что изменилось между alpha-версиями (а их уже
  292) — нет наблюдаемости эволюции API.
- **Решение:** автогенерация CHANGELOG (changesets/conventional-changelog) в релиз-пайплайне.
- **Эффект:** прозрачность изменений, безопасные апгрейды у потребителей.

### O-4. Версия библиотеки недоступна в рантайме
- **Где:** нет экспорта `version`.
- **Серьёзность:** low.
- **Описание:** при дебаге у потребителя нельзя программно узнать версию UI-kit.
- **Решение:** экспортировать `version` (из package.json через сборку).
- **Эффект:** проще диагностика инцидентов («какая версия установлена»).

### O-5. Наблюдаемость сборки/релиза
- **Где:** процесс релиза (см. PL-5).
- **Серьёзность:** low.
- **Описание:** нет следов сборки (логов CI, артефактов, size-отчёта) — релиз
  непрозрачен.
- **Решение:** CI-артефакты: отчёт `size-limit`, coverage, `npm audit`, лог сборки на
  каждый релиз.
- **Эффект:** аудируемая история качества релизов.
