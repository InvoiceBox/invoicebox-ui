# Наблюдаемость

## Примечание про область применения

У этой библиотеки нет собственного трекера ошибок, SDK телеметрии или пайплайна runtime-логирования (подтверждено: нигде в `src/` нет использования Sentry/Bugsnag/LogRocket/аналитики) — и это правильное поведение по умолчанию для UI-кита. Библиотека компонентов не должна в одностороннем порядке решать «звонить домой» в трекер, который потребитель не выбирал. Что здесь оценивается: небольшое количество существующего вывода `console.*`, и чего не хватает, если библиотека когда-нибудь захочет предоставить *хук* для репортинга ошибок, к которому потребители могли бы подключить свой трекер.

## Находки

### 1. 3 вызова `console.error` без структурированного контекста и без способа для потребителя их перехватить
- **Где:** [src/validate/index.ts:163](../src/validate/index.ts) (`"Form dont have vatNumber or countryId"`), [src/components/form/Select/index.tsx:220](../src/components/form/Select/index.tsx) (`"option.entity not found"`), [src/components/form/Select/index.tsx:239](../src/components/form/Select/index.tsx) (`"group.entity not found"`)
- **Серьёзность:** low
- **Почему важно:** это похоже на проверки внутренних инвариантов («эта комбинация props некорректна»), а не на настоящие runtime-ошибки — но сейчас они могут попасть только в консоль браузера, без контекста вроде `endpoint`/`component`/`propName`, и без способа для собственного трекера ошибок приложения-потребителя их увидеть (собственный Sentry потребителя не перехватит голый `console.error`). Если эти случаи представляют собой реальное неправильное использование API, они сейчас невидимы в продакшн-телеметрии.
- **Решение:** два варианта, выбор зависит от намерения: (а) если это задумано как ловля неправильного использования библиотеки потребителем — сделать их выброс исключения только в dev-сборках (громкий сбой на этапе интеграции, а не тихое сообщение в продакшене), которое легко пропустить; (б) если это задумано как представление восстанавливаемых, но заметных состояний рантайма — добавить опциональный проп-колбэк `onError`/`onWarning`, который потребитель может подключить к своему трекеру, чтобы сбои `Select`/`validate` стали видны в том стеке наблюдаемости, который уже есть у приложения-потребителя, вместо того чтобы оставаться только в консоли.
- **Ожидаемый эффект:** эти 3 состояния становятся либо громкими сбоями на этапе разработки (ловит неправильное использование на этапе интеграции), либо видимой потребителю телеметрией (ловит это в продакшене) — сейчас они не то и не другое.

### 2. Нет задокументированного observability-контракта для потребителей
- **Где:** README / эквивалентного документа не существует
- **Серьёзность:** low
- **Почему важно:** если приложение-потребитель хочет узнать «логирует ли эта библиотека что-то / бросает ли исключения, и как к этому подключиться», сейчас на это нет ответа, кроме чтения исходников. Это станет важнее, как только появится колбэк `onError` из находки 1 — его нужно будет задокументировать, а не обнаруживать самостоятельно.
- **Решение:** как только (и если) находка 1 добавит проп-колбэки, задокументировать их в README вместе с короткой заметкой о текущей (правильной) позиции: «эта библиотека не интегрирует напрямую никакой трекер; подключайте `onError` к своему собственному».
- **Ожидаемый эффект:** потребители знают, чего ожидать, не читая исходники `Select`.

## Что уже хорошо

- **Нет шума в консоли за пределами 3 отмеченных вызовов** — нет случайных отладочных `console.log`/`console.warn`, оставленных в кодовой базе, и все 3 существующих — это `console.error` (что корректно является самым серьёзным уровнем из трёх, соответствует принципу из ТЗ «не репортить чрезмерно ожидаемые состояния», даже без формального трекера).
- **Корректно нет интеграции со сторонним трекером** — библиотека компонентов, репортящая собственную телеметрию в захардкоженный трекер (или хуже — в трекер, который потребитель не контролирует), была бы большей проблемой, чем текущий пробел. Отсутствие здесь — правильный дефолт, а не недосмотр, который нужно «чинить» прикручиванием Sentry.
- **`no-console` уже выставлен в `warn`** в [.eslintrc.json](../.eslintrc.json) — какая-то защита от появления console-вызовов уже есть, просто она пока не применяется принудительно, потому что не запускается скрипт `lint` (см. [06-architecture-lint.md](06-architecture-lint.md)).
