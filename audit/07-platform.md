# Платформа

## Находки

### 1. Вообще нет CI — релиз это один локальный bash-скрипт без единой проверки
- **Где:** [release.sh](../release.sh); нет `.github/workflows/`, никакого другого конфига CI нигде в репозитории не найдено
- **Серьёзность:** critical
- **Почему важно:** `release.sh` собирает, коммитит вывод Storybook, поднимает версию, мёржит `develop` → `main` и выполняет `npm publish` — всё одной непрерывной цепочкой, на чьём бы ни было ноутбуке, без **единой** автоматической проверки lint/typecheck/тестов до этого (и скриптов lint/build/typecheck пока даже не существует, чтобы на них опираться — см. [06-architecture-lint.md](06-architecture-lint.md) и [00-PROJECT-MAP.md](00-PROJECT-MAP.md)). Сломанная сборка, падающий тест или нарушение линтера может быть опубликовано в npm, и ничто это не остановит.
- **Решение:** развернуть CI (GitHub Actions, раз репозиторий на GitHub), запускающийся на каждый push/PR в `develop`: установка → lint → typecheck → тесты → сборка. Когда появятся скрипты `lint`/`build`/`typecheck` (это предпосылка из других находок), завязать `release.sh` на прохождение CI для релизимого коммита, либо ещё лучше — перенести саму публикацию в CI-джобу, триггерящуюся по пушу тега, а не в локальный скрипт с доступом к `npm publish`.
- **Ожидаемый эффект:** сломанный коммит больше не может попасть в npm; публикация становится воспроизводимой и аудируемой (кто запустил, что прошло), а не «что было на чьём-то ноутбуке».

### 2. Версия Node на этапе сборки нигде не зафиксирована
- **Где:** [package.json](../package.json) `engines.node: ">=16.0.0"` — и всё; нет `.nvmrc`, нет Dockerfile, нет конфига CI, где можно было бы зафиксировать версию (см. находку 1)
- **Серьёзность:** medium
- **Почему важно:** `>=16.0.0` это нижняя граница, а не фиксация — реальная версия Node, используемая для сборки и публикации, полностью зависит от того, что установлено на машине релизящего в момент запуска. Два разных релиза могли быть собраны с существенно разными мажорными версиями Node без какой-либо записи об этом.
- **Решение:** добавить `.nvmrc` с конкретной версией Node, и когда появится CI (находка 1), использовать ту же зафиксированную версию там через `actions/setup-node`, читающий `.nvmrc`.
- **Ожидаемый эффект:** воспроизводимые сборки — одна и та же версия Node каждый раз, и в CI, и локально.

### 3. `npm audit`: 85 уязвимостей (3 критические, 31 высокая), полностью в дереве devDependency
- **Где:** в основном цепочка `@storybook/preset-create-react-app` → `react-scripts`, плюс прямые devDeps `storybook` и `@babel/core` — полная разбивка в [01-BASELINE-METRICS.md](01-BASELINE-METRICS.md)
- **Серьёзность:** medium (не critical — подтверждено, что ни одна из 5 runtime-зависимостей не задействована, и `files: ["dist"]` означает, что потребителям npm поставляется только собранный вывод, а не само дерево devDependency)
- **Почему важно:** всё же реальная поверхность риска для тех, кто собирает пакет локально или в будущем CI (находка 1), и 3 критических — это не пустяк, даже если ограничено dev-инструментарием.
- **Решение:** это по сути проблема «Storybook 7 отстаёт на 3 мажора от Storybook 10» (см. находку 4) — количество CVE резко упадёт с этим апгрейдом, а не потребует точечного патчинга по одной зависимости. Для всего, что не решится мажорным апгрейдом Storybook, — точечный `npm audit fix`.
- **Ожидаемый эффект:** устраняет 3 critical / 31 high находку без борьбы с lock-файлом по одной зависимости за раз.

### 4. Каждая крупная devDependency отстаёт от актуальной версии на несколько мажоров
- **Где:** `npm outdated` — Storybook 7.6→10.x, ESLint 8→10, `@typescript-eslint` 5→8, TypeScript 4.9 (близка к EOL), React devDep 18.3→19.2 (peer dep намеренно остаётся на `^18.2.0` — это продуктовое решение, а не запущенность, поскольку определяет, какие версии React у потребителей поддерживаются)
- **Серьёзность:** medium
- **Почему важно:** усугубляет находку 3 (CVE в основном живут в устаревшем инструментарии Storybook/Babel) и увеличивает стоимость каждого будущего апгрейда, чем дольше он откладывается — скачки на 3 мажора сложнее, чем ежегодные минорные обновления.
- **Решение:** выделить в отдельную волну (см. [10-REFACTOR-PLAN.md](10-REFACTOR-PLAN.md)) — сначала Storybook 7→10 (самая большая отдача по CVE), затем ESLint 8→10 + `@typescript-eslint` 5→8 вместе (версии друг с другом связаны), затем TypeScript в последнюю очередь (может проявить новые ошибки строгого режима в других местах).
- **Ожидаемый эффект:** меньшая стоимость апгрейдов в дальнейшем; закрывает большую часть счётчика CVE из находки 3 как побочный эффект.

### 5. У `release.sh` нет пути отката, если `npm publish` упадёт на середине
- **Где:** [release.sh:26-40](../release.sh) — поднятие версии, пуш git-тега, мёрдж/пуш `main` и `npm publish` — последовательная цепочка через `&&`
- **Серьёзность:** medium
- **Почему важно:** если `npm publish` падает (сетевой сбой, истекшая авторизация npm, недоступность реестра) *после* того, как версия уже поднята и запушена в git с тегом, репозиторий остаётся в состоянии, где git-тег/версия не совпадают с тем, что реально опубликовано в npm — следующая попытка релиза должна вручную согласовывать этот разъезд.
- **Решение:** либо поменять порядок так, чтобы `npm publish` происходил до пуша git-тега/версии (падать громко до затрагивания состояния git), либо добавить явный шаг отката / заметку в ранбуке в `RELEASE.md` именно для этого сценария сбоя.
- **Ожидаемый эффект:** больше нет молчаливого расхождения версий git/npm после неудачной попытки публикации.

## Что уже хорошо

- **Sourcemaps корректно не публикуются**: ни `tsconfig.json`, ни один из babel-конфигов не включают `sourceMap`/`sourceMaps`, и даже если бы они генерировались, `files: ["dist"]` в `package.json` в сочетании с отсутствием `.map`-файлов означает, что утекать нечему. (Пайплайна загрузки sourcemaps в трекер тоже нет, но и трекера ошибок нет — см. [09-observability.md](09-observability.md) — так что это пока не пробел, просто заметка «когда у библиотеки появится собственный репортинг ошибок, строить пайплайн sourcemaps одновременно».)
- **`dist/` корректно в gitignore и никогда не коммитится** — подтверждено ноль отслеживаемых файлов под `dist/`. Полезная нагрузка npm-публикации полностью остаётся вне истории git, и это правильное решение.
- **Воспроизводимая двойная сборка** уже работает: CJS (`dist/`) и ESM (`dist/esm/`), собранные из одного и того же `src/` через два babel-конфига, отличающихся только настройкой `modules` — чистый, поддерживаемый способ поддержать обе модульные системы без дублирования исходников.
- **`.npmrc` минимальный и чистый** — просто указывает на публичный реестр, без утёкших токенов авторизации или необычных оверрайдов.
