# 06 — Архитектура и линтер

В целом структура здоровая: понятное деление `common`/`form`/`hooks`/`palette`/
`validate`, единый barrel-API, god-компонентов нет (макс 341 LOC). Проблемы точечные.

---

### AR-1. `any` и небезопасные assertions
- **Где:** 11 вхождений `any`/`as any`/`as unknown` по `src` (формы/автокомплит/палитра).
- **Серьёзность:** medium.
- **Описание:** дыры в типобезопасности публичного API библиотеки → ошибки протекают к
  потребителю без подсказок типов. Плюс — `@ts-ignore` = 0 (хорошо).
- **Решение:** заменить `any` на дженерики/`unknown` + сужение; запретить правилами
  `@typescript-eslint/no-explicit-any` (error) и `no-unsafe-*` после апгрейда tooling.
- **Эффект:** надёжные типы на поверхности пакета, меньше runtime-сюрпризов.

### AR-2. Нет ограничения размера файла/сложности в линте
- **Где:** `.eslintrc.json` — нет `max-lines`/`max-lines-per-function`/`complexity`.
- **Серьёзность:** low.
- **Описание:** ничто не сдерживает рост файлов (PhoneInput/Select уже 330–341 LOC).
- **Решение:** добавить `max-lines: [warn, 300]`, `max-lines-per-function`,
  `complexity` как warn; крупные компоненты декомпозировать на под-компоненты/хуки.
- **Эффект:** предсказуемый размер модулей, ранний сигнал о росте сложности.

### AR-3. Лишние ререндеры — необёрнутые объекты/мапы в рендере
- **Где:** напр. `common/AppCardLink/index.tsx` (`appCardMap` создаётся на каждый рендер),
  обработчики/объекты-литералы в JSX по формам.
- **Серьёзность:** low/medium.
- **Описание:** объекты/функции пересоздаются каждый рендер → лишние ререндеры дочерних
  мемоизированных компонентов.
- **Решение:** выносить статические мапы за компонент; `useMemo`/`useCallback` для
  стабильных ссылок там, где есть мемо-потомки. Не мемоизировать вслепую.
- **Эффект:** меньше лишних ререндеров в формах.

### AR-4. 22 `eslint-disable` — точечно скрытый долг
- **Где:** 22 вхождения по `src` (в т.ч. `src/index.ts` — `no-var-requires`/`no-undef`
  для shim-ов `object.entries/fromentries`).
- **Серьёзность:** low.
- **Описание:** часть оправдана (полифилы), часть может маскировать реальные проблемы.
- **Решение:** ревизия каждого disable; shim-полифилы вынести/заменить нативными
  `Object.entries/fromEntries` (доступны в target ES2022) — снять зависимости
  `object.entries`/`object.fromentries` целиком.
- **Эффект:** минус 2 зависимости + чище линт.

### AR-5. Консистентность экспорта/именования
- **Где:** `src/index.ts` — смесь `export type { TProps as … }`; имена палитр
  (`drawer`), пропсы `is*`-флаги.
- **Серьёзность:** low.
- **Описание:** конвенции в целом единые, но `TProps`-переименования и булевы
  `is*`/`should*` стоит зафиксировать в гайдлайне для предсказуемости API.
- **Решение:** короткий CONTRIBUTING/стайлгайд по именованию пропсов и экспортов.
- **Эффект:** предсказуемый, документированный публичный API.

### AR-6. `prop-types` в devDeps у TS-библиотеки
- **Где:** `package.json` devDeps `prop-types`; `react/prop-types` выключен в линте.
- **Серьёзность:** low.
- **Описание:** рудимент — типизация уже на TS; runtime prop-types в `src` не нужны (0).
- **Решение:** удалить `prop-types`, если не используется stories/helpers.
- **Эффект:** чище зависимости.

---

**Приоритет блока:** AR-1 — волна 2; AR-2/AR-4/AR-6 — волна 1 (дёшево, измеримо);
AR-3/AR-5 — волна 3.
